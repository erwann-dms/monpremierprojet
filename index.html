Protocoles de nommage hérités

Les vulérabilités XSS permettent aux attaquants d'injecter du codes malvaillants dans les page web.
Le code executer par la victimes permet aux attaquents de contourner les contrôles d'accès et usurper l'identité des utilisateur. Les failles XSS les plus coourantes sont:
. XSS réfléchi : lorsqu'un utilisateur clique sur une URL malveillante créée par un attaquant, le code JavaScript contenu dans l'URL est exécuté. Comme le code n'est pas stocké sur le serveur,
l'attaquant doit encourager les gens à cliquer sur un lien, en utilisant, par exemple, l'ingénierie sociale.
. XSS stocké : Le code malveillant va être injecté dans une base de données de l'application. Ce code peut être du JavaScript, HTML ou tout autre langage qui peut être interprété par le naviguateur de l'utilisasteur.
L'Exploitations : Dans le cas d'une attaque XSS stockée, une personne malveillante peut insérer un code malveillant dans un commentaire ou un champ de formulaire,
qui sera exécuté lorsqu'un autre utilisateur consultera le contenu.
Une XSS réfléchie se produit lorsqu'un attaquant crée une URL contenant un code malveillant et trompe un utilisateur en l'incitant à cliquer dessus.
Pour ce protéger de cette failles il faut: Filtrage strict des entrées: Tous les données entrées par l'utilisateur doivent être soigneusement filtrées et échappées avant d'être stockées dans la base de données.
Utilisation d'un WAF (Web Application Firewall): Un WAF peut aider à détecter et à bloquer les attaques XSS.
Mise à jour régulière des CMS et des applications: Les vulnérabilités XSS sont souvent corrigées dans les mises à jour de logiciels.
Formation des développeurs: Il est essentiel de former les développeurs aux bonnes pratiques de sécurité pour prévenir les vulnérabilités XSS.
Tests de sécurité réguliers: Des tests de pénétration réguliers permettent d'identifier les vulnérabilités avant qu'elles ne soient exploitées par des attaquants.

Falsification des Requêtes Intersites (CSRF)

Il s'agie d'une attaque qui va permet à l'attaquand de d'incité l'utilisateur athentifié à executer, à sont insu, des action sur une application web.
telles que la réalisation de transactions ou la modification d'informations de leur compte. Cette attaque exploite l'incapacité de l'application web à vérifier l'origine de la requête et la session de l'utilisateur.
Comment fonctionne cette attaque ?:
. Piégeage de l'utilistateur: L'attaquant crée un lien malveillant ou un formulaire caché, trés souvent mit dans une page web inoffensive.
. Executions de la requête: lorsque la victime clique sur ce lien ou visite la pagepiégée, son navigateur envoie automatiquement une requête HTTP à l'application web cible.
Cette requête est conçue pour exécuter une action spécifique, comme la suppression d'un compte, le transfert de fonds ou la modification de paramètres.
. Exécution de l'action: Si l'application ne vérifie pas l'origine de la requête et s'appuie uniquement sur la session de l'utilisateur pour autoriser l'action,
celle-ci sera exécutée, sans que la victime n'en ait conscience.
Comment ce protéger:
Vérfications de l'origine de la requête: S'assurer que la requête provient bien de votre domaine.
HTTPOnly cookies: Empêcher le JavaScript client d'accéder aux cookies de session.
Méthode POST: Privilégier la méthode POST pour les actions sensibles.
Mise à jour régulière des applications: Corriger les vulnérabilités connues.

Mauvaise configuration CORS

Par défaut, les applications ne peuvent pas communiquer en JavaScript avec des applications provenant d'autres domaines. Le partage de ressource cross-origin (CORS - Cross-Origin Resource Sharing) peut être utilisé pour modifier ce comportement.
Une mauvaise configuration peut rendre l'application vulnérable à un accès non autorisé depuis d'autres domaines.
Comment est elle exploitée ?:
Les erreurs de configuration les plus fréquentes sont la mauvaise configuration de « Access-Control-Allow-Origin », qui, si elle est trop permissive,
utilise un joker (*) ou la valeur null, permet d'effectuer des requêtes à partir de domaines non contrôlés. Si, en plus, l'application utilise « Access-Control-Allow-Credentials : True »,
un attaquant pourrait effectuer des requêtes à partir des sessions des utilisateurs, et ainsi voler leurs comptes, car contrairement aux attaques CSRF, l'attaquant peut lire les résultats de ces requêtes. Comment ce protéger ?:
Évitez d'utiliser des joker ou la valeur null dans l'en-tête Access-Control-Allow-Origin. Limitez l’entête CORS aux domaines de confiance

Référence Directe d'Objet non sécurisé

Il s'agie d'une vulnérabilité qui est courante dans les appliction web qui survient lorsque l'application utilise une entrée fournie directement par l'utilisateur pour accéder à un objet.
(un fichier, une entrée de base de données,ect).Les attaquents pouvent manipuler des référence internes pour accéder à des données non autorisées. Pour faire simple l'attaquand peut manipuler l'identifiant de l'objet dans l'URL ou dans un formulaire pour accéder à des données.
Comment elle fonctionne ?:
prenons un exemple simple : un site e-commerce. Chaque produit est identifié par un ID unique dans l'URL. Un attaquant pourrait simplement incrémenter cet ID dans l'URL,
pour accéder à des produits qui ne lui sont pas destinés, voire à des données internes de l'application.
URL normale : https://monsite.com/produit/123
URL exploitée : https://monsite.com/produit/124

Vulnérabilité sur le téléchargement d'un fichier

Permettre aux utilisateurs de téléverser des fichiers peut présenter des risques pour la sécurité si ce n’est pas correctement géré.
magine un site web qui te permet de télécharger des photos. Cette fonctionnalité, bien pratique, peut devenir un véritable point faible si elle n'est pas correctement sécurisée.
C'est ce qu'on appelle une vulnérabilité de téléchargement de fichiers.
Commant çe la foctionne
Les attaquants peuvent exploiter cette vulnérabilité en téléchargeant des fichiers qui ne sont pas de simples images, mais des fichiers malveillants déguisés. Ces fichiers peuvent contenir :
. Des virus ou des logiciels malveillants qui vont infecter ton ordinateur ou le serveur du site.
. Du code qui permet de prendre le contrôle du serveur et d'accéder à toutes les données qui s'y trouvent.
. Des scripts qui peuvent être exécutés sur le site web pour voler des informations ou défigurer les pages.
Comment ce protéger? : . Limiter les types de fichiers autorisés: Ne permettre le téléchargement que de certains formats de fichiers (images, PDF, etc.) et vérifier leur extension.
. Scanner les fichiers à la recherche de virus: Utiliser un antivirus pour analyser les fichiers téléchargés avant de les stocker.
. Limiter la taille des fichiers: Fixer une taille maximale pour les fichiers téléchargés afin d'éviter les attaques par surcharge.
. Stocker les fichiers dans un répertoire sécurisé: Isoler les fichiers téléchargés des autres fichiers du serveur pour limiter les dommages en cas d'attaque.
. Utiliser des noms de fichiers aléatoires: Éviter d'utiliser les noms de fichiers originaux pour rendre plus difficile l'exploitation de certaines vulnérabilités.

Redirection ouverte

Il s'agie d'une attaque que quand t'on sera sur un site que si on cliques sur un lien. Au lieu d'être redirigé vers la page prévue, on ce retrouves sur un site complètement différent, peut-être même dangereux.
C'est ce qui peut se produire lorsqu'un site web souffre d'une vulnérabilité de redirection ouverte.
Comment cela fonctionne ?
L'utilisateur entre une URL : Le site web propose un champ où tu peux entrer une URL.
Le site redirige : Quand on cliques sur un bouton, le site te redirige directement vers l'URL que on as saisie, sans vérification.
L'attaquant en profite : Un hacker malintentionné peut te fournir une URL malveillante, et lorsque on à cliquer dessus, tu seras redirigé vers un site frauduleux.
Comment ce protéger:
Validation des URL: Le site web doit vérifier que l'URL saisie est bien formatée et qu'elle pointe vers un domaine autorisé.
Filtrage des URL: Certaines chaînes de caractères peuvent être interdites dans les URL pour empêcher les attaques.

Vulnérabilité des JWT

Un Jeton Web JSON (JWT) est un format standard ouvert utilisé pour la transmission d'informations de manière sécurisée entre différentes parties d'une application.
Il est souvent utilisé pour l'authentification, mais peut également servir à stocker des données utilisateur. Mais une mauvaise implémentation peut entraîner des risque de sécurité.
Les risque de un attaques:
Vol du JWT: Si un JWT est intercepté, un attaquant pourrait usurper l'identité de l'utilisateur.
Falsification du JWT: Un attaquant pourrait modifier les informations contenues dans un JWT pour obtenir des privilèges supplémentaires.
Expiration du JWT: Si la durée de vie d'un JWT est trop longue, cela augmente le risque d'être compromis.
Stockage non sécurisé: Stocker un JWT dans le localStorage ou le sessionStorage du navigateur peut exposer l'information à des attaques XSS.
Réutilisation de JWT: Réutiliser un JWT peut permettre à un attaquant de se connecter en tant qu'un autre utilisateur.
Comment sécuriser le JWT ?
Choisir un algorithme de signature fort: Utiliser des algorithmes de signature cryptographiques robustes
Limiter la durée de vie des JWT: Fixer une durée de vie courte pour les JWT afin de minimiser les risques en cas de compromission.
Stocker les JWT de manière sécurisée: Éviter de stocker les JWT dans le localStorage ou le sessionStorage. Privilégier le stockage côté serveur.
Protéger la clé secrète: La clé secrète utilisée pour signer les JWT doit être strictement confidentielle.
Vérifier les revendications: Toujours vérifier les revendications contenues dans un JWT avant d'accorder des accès.

Composants présentant des vulnérabilités connues

Un composant vulnérable, dans le contexte des applications web, désigne généralement une bibliothèque, un framework ou un module tiers intégré à une application.
Ce composant contient une faille de sécurité connue qui peut être exploitée par des attaquants.
Pourquoi elle est dangereux ?
Porte d'entrée pour les attaquants: Ces vulnérabilités offrent aux attaquants une entrée directe dans l'application, leur permettant d'exécuter du code arbitraire,
d'accéder à des données sensibles ou de prendre le contrôle du système.
Amplification des attaques: Une vulnérabilité dans un composant largement utilisé peut affecter un grand nombre d'applications, augmentant ainsi la surface d'attaque globale.
Difficulté de mise à jour: Il peut être difficile de mettre à jour tous les composants d'une application, surtout si certaines dépendances sont obsolètes ou si l'application est complexe.
Comment ce protégée:
Choisir des composants fiables: Privilégier les composants maintenus par une communauté active et qui font l'objet de nombreux audits de sécurité.
Mettre à jour régulièrement: Mettre en place un processus de mise à jour automatique ou semi-automatique des composants.
Utiliser des outils de gestion des dépendances: Ces outils permettent de gérer les différentes versions des composants utilisés dans un projet et de détecter les vulnérabilités.
Effectuer des analyses de vulnérabilité: Utiliser des outils de scan pour détecter les vulnérabilités dans les applications et les composants.
Isoler les composants: Si possible, isoler les composants les plus sensibles dans des conteneurs ou des environnements virtuels.
Former les développeurs: Sensibiliser les développeurs aux risques liés aux composants vulnérables et leur apprendre les bonnes pratiques.

Stockage non sécurisé des jetons de session

Un jeton de session est un identifiant unique, généralement un long texte aléatoire, qui est attribué à un utilisateur
lorsqu'il se connecte à un service en ligne.
Ce jeton est ensuite envoyé avec chaque requête ultérieure et permet au serveur d'identifier l'utilisateur et de maintenir sa session active.
un attaquant peut détourner la session, usurper l'identité de l'utilisateur et accéder à des informations sensibles.
Quelle sont les risque ?
. Vol de session: Un attaquant qui parvient à intercepter un jeton de session valide peut se faire passer pour
l'utilisateur légitime et accéder à toutes ses données et fonctionnalités.
. Fixation de session: Un attaquant peut forcer un utilisateur à utiliser un jeton de session qu'il contrôle,
lui permettant ainsi de prendre le contrôle du compte de l'utilisateur.
Comment stooker en sécuritée le jalon ?
. Chiffrement: Les jetons de session doivent toujours être chiffrés avant d'être stockés. . Durée de vie limitée: Les jetons doivent avoir une durée de vie limitée et être automatiquement
invalidés après un certain temps d'inactivité. . Stockage sécurisé: Les jetons ne doivent jamais être stockés en clair dans des fichiers de configuration
ou des bases de données accessibles publiquement
. HTTPOnly: Les cookies contenant les jetons de session doivent être marqués comme "HttpOnly" pour empêcher leur accès
par des scripts côté client. .Secure: Les cookies doivent être transmis uniquement via des connexions HTTPS pour éviter les interceptions.
CSRF protection: Des mesures de protection contre les attaques CSRF doivent être mises en place pour empêcher la falsification de requêtes inter-sites.

Injection SQL

Il s'agie de une technique d'attaque informatique qui exploite une vulnérabilité dans la manière dont les applications web gèrent les entrées utilisateur.
cette attaque permettent aux attaquants d'exécuter des requêtes malveillantes en injectant du code SQL dans les champs d'entrée.
Quelle sont les risque d'une attaque SQL ?
. Vol de données: L'attaquant peut accéder à des données sensibles stockées dans la base de données, telles que des mots de passe,
des numéros de carte de crédit ou des informations personnelles.
. Modification de données: L'attaquant peut modifier ou supprimer des données de la base de données,
ce qui peut avoir des conséquences désastreuses pour l'entreprise. . Dénis de service: L'attaquant peut lancer des attaques par déni de service en surchargeant la base de données avec des requêtes malveillantes.
. Privilège d'escalade: Dans certains cas, l'attaquant peut obtenir des privilèges élevés sur le serveur de base de données et compromettre
l'ensemble du système.
Comment ce protèger ?
. Paramétrisation des requêtes: Utilisez des requêtes préparées ou des paramètres pour éviter la concaténation de chaînes de caractères.
. Échappement des caractères spéciaux: Filtrez et échappez tous les caractères spéciaux qui pourraient être utilisés pour manipuler la syntaxe SQL.
. Validation des entrées: Vérifiez toujours la validité et le type des données fournies par l'utilisateur.
. Utilisation d'un ORM: Un Object-Relational Mapper (ORM) peut aider à prévenir les injections SQL en générant automatiquement des requêtes sécurisées.
. Mise à jour régulière des logiciels: Maintenez vos applications et votre base de données à jour avec les derniers correctifs de sécurité.
